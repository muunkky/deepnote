# Design Doc: Phase 1 — Run a single-language non-Python notebook end-to-end via `--kernel`

> **ADRs**: ADR-002, ADR-003, ADR-004 | **Date**: 2026-06-11 | **Author**: muunkky

## Overview

This design implements **PRD-002 Phase 1**: a user runs `deepnote run --kernel <name> notebook.deepnote` and a single-language, non-Python notebook (plain code + markdown only) executes end-to-end against an already-installed Jupyter kernel, returning MIME-typed outputs. Three accepted ADRs together define the slice. ADR-002 (launch model) decided that non-Python execution is a kernelspec-_selection_ problem, not a new-runtime problem: the existing `python -m deepnote_toolkit server` already hosts any registered kernelspec (proven by the `docs/spikes/nom-002/` spike), so the only code blocker is the hardcoded `kernel: { name: 'python3' }` at `kernel-client.ts:78`. ADR-003 (selection source) decided the precedence `explicit --kernel > notebook-declared language (Phase 2, forward-declared) > 'python3'`, on a channel separate from `DEEPNOTE_PYTHON`, resolved by a dedicated pure selector mirroring `selectPythonSpec`. ADR-004 (degradation) decided that value-add blocks (SQL, visualization, DataFrame-formatting, input, agent) **hard-fail** with a typed, entity-naming error on a non-Python kernel, and that reactivity is **bypassed** (never attempted) on the only paths that invoke it.

The decisions are made; this doc designs the _how_. The work threads one new value (`kernelName`) through three existing seams — the CLI option layer (`cli.ts` / `run.ts`), `RuntimeConfig` (`types.ts`), and the `@jupyterlab/services` session-start call (`kernel-client.ts`) — plus a pre-flight `GET /api/kernelspecs` validation that converts the server's opaque `HTTP 500` into a typed `KernelNotRegisteredError`, a non-Python predicate that gates two degradation guards, a configurable idle-timeout so heavy kernels are not failed spuriously, and a machine-readable failure-category field on `RunResult` that is populated at **both** the success-summary path and the run-time error path (see KD-6). With `kernelName` defaulting to `'python3'`, every existing notebook takes a byte-stable, round-trip-free path; non-Python is purely additive.

The single largest engineering risk is **test reality**: CI today installs no Python and runs zero real-server execution tests — `KernelClient`, `ExecutionEngine`, and `startServer` are fully mocked (`vi.mock`/`vi.hoisted`), and the only "e2e" job runs `node dist/bin.js --version|--help`. The headline launch criterion (a non-Python kernel running through the real CLI and returning a non-`text/plain` MIME bundle) therefore cannot run in the current CI. This doc specifies the IaC for it explicitly: a new, dedicated CI job that provisions Python + `deepnote-toolkit[server]` + `bash_kernel` (reusing the `docs/spikes/nom-002/` harness), collected by a separate vitest project (the default config gains an `exclude` glob — see R8/KD-9), so the design is built against the reality the research found rather than an assumed one.

## Requirements

The implementation is complete when:

1. **R1 (ADR-002):** `RuntimeConfig` carries an optional `kernelName` field defaulting to `'python3'`; `KernelClient.connect(serverUrl, kernelName?)` passes `kernel: { name: kernelName }` to `SessionManager.startNew` in place of the literal at `kernel-client.ts:78`; and `ExecutionEngine` forwards `config.kernelName`. With no kernel set, the resolved name is `'python3'` and the Python path is behaviorally identical to today (existing mocked suites pass unchanged).
2. **R2 (ADR-002):** Before `startNew`, for any explicitly-set non-`python3` `kernelName`, the runtime issues `GET {serverUrl}/api/kernelspecs`; an absent name raises a typed `KernelNotRegisteredError` carrying the requested name and the available `name → {display_name, language}` list **before** any `POST /api/kernels` — so the opaque server `500` never reaches the caller. The pre-flight is **skipped for the literal `python3` default** (Python path stays round-trip-free). A failed pre-flight GET falls back to the existing `waitForServer` `/api` readiness handling, not a missing-kernel error.
3. **R3 (ADR-003):** A pure `selectKernelName({ explicit })` selector in runtime-core resolves `explicit ?? 'python3'` (the Phase-2 declared tier is forward-declared but not wired); a `--kernel <name>` option on the `run` command feeds the `explicit` tier; the selector reads no env (a test asserts `DEEPNOTE_PYTHON` does not affect it); and the resolved kernel is echoed in CLI human output.
4. **R4 (ADR-004):** When the active kernel is non-Python, execution reaching a value-add block (SQL / visualization / DataFrame-formatting / input / agent) aborts the run at that block with a typed `UnsupportedBlockOnKernelError` naming the block type and the kernel, and **no `_dntk.*` Python string is dispatched to the kernel**. Plain code + markdown run normally on the selected kernel.
5. **R5 (ADR-004, efficiency/honesty — see KD-5 for the corrected premise):** On the analyzer-invoking paths only (`--block` / dag / analyze / lint, past the `if (!options.block)` early-return at `run.ts:444`), a non-Python kernel **bypasses** the `getUpstreamBlocks` call entirely — executing requested blocks in existing order with a "reactivity is Python-only" notice — so the runtime does **not** spawn a Python AST subprocess (`ast-analyzer.py`) that is known in advance to fail on non-Python source, and the user sees an honest notice instead of silent dependency-less execution. The whole-notebook path's `validateRequirements` is given the **same** bypass (it too calls the Python analyzer; see the should-address correction in KD-5). The observable contract of this requirement is _that the analyzer is not invoked_, not "no `AstAnalyzerInternalError` reaches the user" (the latter is already true today because both call sites try/catch and degrade gracefully — see R5's test design).
6. **R6 (PRD-002 CI criterion):** The four kernel-failure classes — missing kernelspec, kernel-launch failure, kernel-died-mid-run, in-block runtime error — are each distinguishable on a **stable machine-readable field** in `--output json` (a new `failureCategory` on `RunResult`); no two classes collapse to one value. The field is populated at **two** surfacing sites because three of the four classes never reach the success-path summary builder (see KD-6 and the Interface Design "two surfacing sites" subsection): (a) the run-action outer catch (`run.ts:522`) for `missing-kernel`, `kernel-launch`, and launch-time `kernel-died`; (b) `buildMachineRunResult` (`run.ts:1212`) for `kernel-died`-mid-run and `in-block`. Because the CLI's `onBlockDone` (`run.ts:1058`) flattens the engine's typed `Error` instance to its `.message` string before it reaches `state.blockResults` (`BlockResult.error: string | undefined`, `run.ts:105`), site (b) does **not** rely on an `instanceof` of a stringified error; instead it reads an explicit `failureCategory` discriminant added to `BlockResult` (and to `RunExecutionState.blockResults`), computed at the serialization point `run.ts:1058` from the still-typed `result.error` (`instanceof KernelDiedError → 'kernel-died'`, else `'in-block'`). (Exit-code subdivision is explicitly _not_ done; see Key Design Decisions.)
7. **R7 (PRD-002 headline launch criterion, TDD):** An integration test runs a pure-pip non-Python kernel (`bash_kernel`) end-to-end through the **real CLI** against the **real** `deepnote-toolkit` server with `--kernel bash`, and asserts a **non-`text/plain` MIME bundle (`image/png`)** comes back — exercising the JSON-only WebSocket fallback at `kernel-client.ts:18-41` and IOPub decode against a binary-output kernel. This test runs in a dedicated, Python-provisioning CI job (R8), collected by a separate vitest project, not the always-on mocked suite.
8. **R8 (test/CI reality):** A new CI job provisions Python 3 + `deepnote-toolkit[server]` + `bash_kernel` (reusing `docs/spikes/nom-002/` setup) and runs the integration suite via a dedicated `test:integration` script backed by a separate vitest project. The default vitest config gains an explicit `exclude: ['**/*.integration.test.ts']` glob so the integration files are **not** collected by the always-on mocked `pnpm test` (the default config does change — see KD-9, correcting the prior "unchanged" claim); `RUN_INTEGRATION_TESTS=true` is a defense-in-depth runtime gate, not the exclusion mechanism. The always-on mocked Node suite still gates every commit; cspell gains only the genuinely-missing kernel terms (`bash_kernel`, `kernelName`, `kernelspecs`, `IJulia`, `IRkernel`) — `kernelspec` and `iopub` are already present in `cspell.json` and are **not** re-added.

## Current State

Concrete, as verified in the research:

- **Hardcoded kernel name.** `KernelClient.connect(serverUrl)` (`kernel-client.ts:55`) takes only a URL; line 78 hardcodes `kernel: { name: 'python3' }` inside `SessionManager.startNew`. This is the **only** site a kernel name is specified.
- **`RuntimeConfig` has no kernel field.** `types.ts:3-12` carries `pythonEnv`, `workingDirectory`, optional `serverPort`, optional `env` — nothing kernel/language.
- **`ExecutionEngine` never threads a kernel name.** `constructor(config: RuntimeConfig)` (`execution-engine.ts:101`) stores config; `start()` (`:113-127`) calls `startServer(...)` then `new KernelClient()` (`:122`) and `kernel.connect(this.server.url)` (`:123`). No kernel name flows. `KernelClient` is held as a private field with **no accessor for kernel state**, and `connect()` returns `void` (`kernel-client.ts:55`) — relevant to KD-3's language hand-off.
- **No `--kernel`, no kernel selector.** `cli.ts:274` defines only `--python`; `run.ts` `RunOptions` (`:79-95`) has no kernel field. Python resolution uses the pure `selectPythonSpecWithHint` (`python-env.ts:177-224`); there is no `selectKernelName` analogue. `ExecutionEngine` is instantiated at `run.ts:864-867` with only `{ pythonEnv, workingDirectory }`.
- **`createPythonCode` is unconditional.** `execution-engine.ts:375` calls `createPythonCode(block)` for every executable block and pipes the string to `kernel.execute(code)`; the agent branch opens at `:236` (`if (isAgentBlock(block))`) and closes at the `} else {` before the dispatcher at `:374`. The dispatcher (`python-code.ts:29-91`) emits `_dntk.*` RPC for SQL/viz/DataFrame and Python literals for inputs; markdown/text/image/separator never reach it (filtered by `isExecutableBlock`, `execution-engine.ts:197`). It throws `UnsupportedBlockTypeError extends DeepnoteError` (`errors.ts:53`, thrown `python-code.ts:90`) for genuinely-unknown types — the typed-error pattern to mirror.
- **Per-block error preservation — but the typed class is FLATTENED to a string at the CLI boundary.** The per-block execution loop in the engine catches at `execution-engine.ts:399-413` and stores the thrown `Error` _instance_ on `BlockExecutionResult.error` (`:410`), then `break`s — so the engine layer still holds the typed class (e.g. `KernelDiedError` vs. a user `NameError`). **However, that instance does not survive into the CLI's `state.blockResults`:** the CLI's `onBlockDone` callback (`run.ts:1048-1059`) pushes a `BlockResult` whose `error` is set to `result.error?.message` (`run.ts:1058`) — a **string**. `BlockResult.error` is typed `string | undefined` (`run.ts:105`), and `RunExecutionState.blockResults` is `BlockResult[]` (`run.ts:193-194`). `buildMachineRunResult` (`run.ts:1212-1226`) receives that `BlockResult[]`, so an `instanceof KernelDiedError` check against `blockResults[].error` can **never** be true — `kernel-died`-mid-run would silently collapse into `in-block`, violating R6. The discriminant must therefore be captured **at the serialization point** (`run.ts:1058`), where the typed `result.error` instance is still in scope, and carried forward as an explicit field (see KD-6 site (b)).
- **Executable-type set lives in `packages/blocks`.** `packages/blocks/src/blocks/executable-blocks.ts` defines `INPUT_BLOCK_TYPES` (`:7-16`) and a **private** `executableBlockTypes` Set (`:18-27`) plus `isExecutableBlock`/`isExecutableBlockType` (`:33-43`). The `executableBlockTypeSet`/`executableBlockTypes` names referenced elsewhere in the CLI are a **re-export shim in runtime-core** at `execution-engine.ts:26-44` ("Re-export for backwards compatibility - these are now defined in @deepnote/blocks"). The canonical set is the blocks one; the runtime-core copy is a list literal kept in sync.
- **Reactivity gating — and both analyzer call sites already degrade gracefully.** `resolveUpstreamExecutionBlockIds` (`run.ts:439-484`) returns at `if (!options.block) return undefined` (`:444`) before the analyzer call `getUpstreamBlocks` (`:459`). So whole-notebook `deepnote run` does not enter _this_ function's analyzer path; `--block`/dag/analyze/lint do. **`getUpstreamBlocks` is imported from `@deepnote/reactivity`** (`run.ts:8`), defined at `packages/reactivity/src/dag.ts:116`; it wraps the whole analysis in `try/catch` and returns `{ status: 'fatal', error }` on **any** error (`dag.ts:138-143`) — it never re-throws `AstAnalyzerInternalError`. `resolveUpstreamExecutionBlockIds` already handles `status === 'fatal'` by logging and returning `undefined` (`run.ts:463-466`), running the single block without deps. **Separately, the whole-notebook path _does_ analyze:** `validateRequirements` (`run.ts:755`) calls `getBlockDependencies` (`run.ts:793`) — the same Python AST subprocess (`ast-analyzer.py` via spawn) — inside a `try/catch` that on failure logs and `return`s, skipping input validation (`run.ts:794-798`). So `AstAnalyzerInternalError` **cannot surface to the user today on any path**; both sites already swallow it. The Phase-1 change is therefore an efficiency/honesty change (don't spawn a subprocess known to fail; emit an explicit notice), not a crash fix.
- **Undifferentiated failures, and the error-path JSON has no category.** `kernel-client.ts:82-120` throws only `new Error('Failed to start kernel')`, `'Kernel is dead'`, and a fixed `waitForKernelIdle(timeoutMs = 30000)` (`:97`) timeout. Kernel-lifecycle errors (preflight `KernelNotRegisteredError`, `startNew` rejection, launch-time death) propagate out of `engine.start()` and are caught by the **outer catch at `run.ts:522-543`**, which emits `outputJson({ success: false, error: message })` (`:534`) with **no** `failureCategory` field. `buildMachineRunResult` (`run.ts:1212-1317`) is only called on the **success return path** (`run.ts:902-910`), after `engine.runProject` returns a summary — so it only ever sees `in-block` and `kernel-died`-mid-run failures via `state.blockResults`. `RunResult` (`run.ts:156-164`) carries only boolean `success`. `ExitCode` is exactly `Success=0`/`Error=1`/`InvalidUsage=2` (`exit-codes.ts:11-22`).
- **Test/CI reality.** Vitest 4.1.8, all **unit** tests; root `vitest.config.ts` uses `include: ['**/*.test.ts']` with `bail: 1` and **no** `exclude`. `execution-engine.test.ts` and `kernel-client.test.ts` fully mock `KernelClient`, `startServer`, and `@jupyterlab/services`. There is **no `test:integration` script and no vitest projects config today** (verified). CI (`.github/workflows/ci.yml`) runs spell-check/lint/typecheck/build/test/cli-e2e on **Node only** (22.21.0 + 22/23/24/25); **no `actions/setup-python`, no venv, no toolkit, no real kernel**. The `cli-e2e` job (`:198-228`) only runs `--version`/`--help`. The reusable real-server harness is `docs/spikes/nom-002/` (`probe.py` + `SPIKE-FINDINGS.md`/`RESULTS.md`: `python3 -m venv venv; ./venv/bin/pip install "deepnote-toolkit[server]" bash_kernel jinja2; ./venv/bin/python -m bash_kernel.install --sys-prefix`). cspell (`cspell.json` + `docs-dictionary.txt`) **already** has `kernelspec` (`cspell.json:75`) **and** `iopub` (`cspell.json:66`); the genuinely-missing terms are `bash_kernel`, `kernelName`, `kernelspecs`, `IJulia`, `IRkernel` (none appear in `cspell.json` or `docs-dictionary.txt`, verified).

The kernel name is **forwarded, not stored per-session in two places**: `RuntimeConfig.kernelName` is the single owner; `connect()` receives it as a parameter (defaulting to `'python3'` so the existing one-arg `connect(url)` call sites and tests stay valid until updated). This mirrors how `pythonEnv` rides `RuntimeConfig` and the engine threads it. The kernelspec **language** flows back the other direction: `connect()` returns it (see KD-3) and the engine stores it on a private field, so the engine never re-fetches `/api/kernelspecs`.

### Key Design Decisions

**KD-1 — `selectKernelName` is a separate pure resolver, not an overload of `selectPythonSpec`.** ADR-003 Key Factor 4 mandates a distinct selector so the two axes are independently unit-testable and cannot entangle. It is a `string | undefined → string` precedence function with no I/O and no env read:

```ts
// packages/runtime-core/src/kernel-name.ts
export const DEFAULT_KERNEL_NAME = "python3";

export interface SelectKernelNameOptions {
  /** Highest-precedence tier: an explicit --kernel value for this run. */
  explicit?: string;
  /**
   * Phase-2 tier (forward-declared, not wired now): the notebook-declared
   * language already mapped to a kernel name. Undefined in Phase 1.
   */
  declared?: string;
}

export function selectKernelName({
  explicit,
  declared,
}: SelectKernelNameOptions): string {
  // explicit ?? declared ?? default — empty/whitespace treated as absent,
  // mirroring selectPythonSpec's fall-through.
  const pick = (v?: string) =>
    v && v.trim().length > 0 ? v.trim() : undefined;
  return pick(explicit) ?? pick(declared) ?? DEFAULT_KERNEL_NAME;
}
```

_Rationale:_ exact isomorphism with `selectPythonSpec`'s no-assembly shape (`python-env.ts:160-162`) — callers learn one mental model. _Alternatives rejected:_ a `selectKernelNameWithHint` analogue (ADR-001's hint exists because bare-system-Python needs an onboarding nudge; the kernel axis has no equivalent ambient default to warn about — the missing-kernel _case_ is owned by R2's pre-flight, not a selector hint); and a `DEEPNOTE_KERNEL` env tier (ADR-003 defers it explicitly — documented in the resolver's doc comment, not built).

**KD-2 — pre-flight skips the GET for literal `python3`; runs for any other explicit name.** ADR-002 Decision point 4 offers skip-for-`python3` vs. uniform and requires picking one; we pick **skip-for-`python3`** so the overwhelmingly-common Python path stays round-trip-free and byte-identical to today. The predicate is `kernelName === DEFAULT_KERNEL_NAME`, not "looks Pythonish" — exact-match is safe per ADR-002 Implementation Notes (the toolkit's _reported_ default `python3-venv` is a red herring; literal `python3` always resolves).

**KD-3 — Phase 1 uses the name-based non-Python predicate; kernelspec `language` is captured but forward-looking.** ADR-004 Implementation Notes offer a name-based ("anything but `python3`") predicate vs. a `language`-based one. The two reactivity-bypass guards (KD-5) run **pre-connect** — before the engine starts the server, so no `/api/kernelspecs` map and no `language` exists at those sites; they _must_ use the name-based check. For consistency, the per-block value-add guard (R4) also uses the name-based predicate in Phase 1, which is sufficient for the Phase-1 target: `bash` is non-`python3`, so `isNonPythonKernel('bash')` returns `true` with no language. The kernelspec `language` _is_ captured during pre-flight (it is free — the `/api/kernelspecs` map we already fetch for R2 carries `{display_name, language}`, spike finding 6) and is **returned from `connect()`** so the engine can store it, but it serves as a forward-looking signal for Phase 3 `--list-kernels` discovery and as an optional refinement, not as the load-bearing Phase-1 source of truth. `isNonPythonKernel` therefore takes `language` as an _optional_ refinement: when present it tightens the decision, when absent the name decides.

```ts
export function isNonPythonKernel(
  kernelName: string,
  language?: string,
): boolean {
  if (kernelName === DEFAULT_KERNEL_NAME) return false; // python3 fast-path
  if (language) return language.toLowerCase() !== "python"; // optional refinement (Phase 3+)
  return true; // Phase-1 load-bearing path: any explicit non-python3 name => non-Python
}
```

The concrete language hand-off: `connect(serverUrl, kernelName?)` returns `Promise<string | undefined>` (the resolved kernelspec language, or `undefined` for the `python3` fast-path / failed GET); `ExecutionEngine.start()` stores it on a private `kernelLanguage` field and passes it to `isNonPythonKernel(this.config.kernelName, this.kernelLanguage)` at the per-block seam. No new accessor on `KernelClient` is needed — the value rides the existing `connect()` return.

**KD-4 — value-add classification set lives beside the canonical `executableBlockTypes` in `packages/blocks`.** ADR-004 Implementation Notes name the seam: at/just-before `createPythonCode(block)` (`execution-engine.ts:374-378`) plus the agent branch (`:236`). We add a `VALUE_ADD_BLOCK_TYPES` set in **`packages/blocks/src/blocks/executable-blocks.ts`**, next to the private `executableBlockTypes` Set (`:18-27`) and `INPUT_BLOCK_TYPES` (`:7-16`) — the package that owns block-type knowledge. (Note: the `executableBlockTypeSet` symbol the CLI uses is the **runtime-core re-export shim** at `execution-engine.ts:26-44`, kept in sync with the blocks definition; the new set belongs with the canonical definition, not the shim.) `VALUE_ADD_BLOCK_TYPES` is defined as `executableBlockTypes` **minus `'code'`** (and excluding non-executable `markdown`, which never enters the executable set) — i.e. every executable type whose codegen emits `_dntk.*` or Python literals: `sql`, `visualization`, `big-number`, `button`, `notebook-function`, the `input-*` types (via `INPUT_BLOCK_TYPES`), and `agent`. The engine, when `isNonPythonKernel` is true, tests membership _before_ calling `createPythonCode`/`executeAgentBlock` and throws — so **no `_dntk` string is ever generated, let alone dispatched** (satisfies R4's "assert no Python RPC dispatched"). A unit test cross-checks `VALUE_ADD_BLOCK_TYPES` coverage against the `python-code.ts:29-91` dispatcher so a new value-add type cannot silently slip through. _Rejected:_ a new `block.requiresPython` schema field or a `BlockCategory` enum (research gap) — that is new architecture the ADRs do not own; a set derived from existing `.type` literals tracks the dispatcher with zero schema change, which ADR-004 explicitly calls "a design-doc artifact that must track the dispatcher."

**KD-5 — reactivity is bypassed at _both_ Python-analyzer call sites; this is an efficiency/honesty change, not a crash fix.** The prior draft's premise was wrong on two counts, corrected here:

1. _Not a crash fix._ `getUpstreamBlocks` (`@deepnote/reactivity`, `dag.ts:116`) already wraps analysis in `try/catch` and returns `{status:'fatal', error}` (`dag.ts:138-143`); `resolveUpstreamExecutionBlockIds` already handles `'fatal'` by returning `undefined` (`run.ts:463-466`). `validateRequirements` already `try/catch`es `getBlockDependencies` and `return`s on failure (`run.ts:794-798`). So `AstAnalyzerInternalError` **cannot reach the user today on any path** — "no `AstAnalyzerInternalError` surfaced" is already true and cannot drive the code. The ADR-004 rationale that _does_ hold is efficiency + honesty: don't spawn a Python AST subprocess (`ast-analyzer.py`) that is known in advance to fail on non-Python source, and tell the user explicitly that reactivity is Python-only rather than silently running dependency-less.

2. _The whole-notebook path also analyzes._ The prior claim that the whole-notebook path "never analyzes / stays analyzer-free by construction" was false: `validateRequirements` (`run.ts:755`, invoked on the whole-notebook run) calls `getBlockDependencies` → the same Python subprocess. ADR-004's narrower true claim was only about `resolveUpstreamExecutionBlockIds`/`getUpstreamBlocks`. Both sites get the bypass.

Implementation: add a **name-based** non-Python check (these sites run pre-connect, so `language` is unavailable). Thread the resolved `kernelName` into both functions — neither takes it today: `resolveUpstreamExecutionBlockIds(file, options, pythonInterpreter)` (`run.ts:439`) and `validateRequirements(file, providedInputs, pythonInterpreter, integrations, notebookName?)` (`run.ts:755`) gain a `kernelName: string` parameter. In `resolveUpstreamExecutionBlockIds`, after the `if (!options.block) return undefined` early-return (`:444`) and before `getUpstreamBlocks` (`:459`): if `kernelName !== DEFAULT_KERNEL_NAME`, emit the "reactivity is Python-only; running without dependency analysis" notice and `return undefined` — reusing the existing `fatal`-branch ordered-execution fallback shape (`:463-466`), so `getUpstreamBlocks` is never called. In `validateRequirements`, before `getBlockDependencies` (`:793`): if `kernelName !== DEFAULT_KERNEL_NAME`, emit the same notice and skip the call (the existing `try/catch` remains as a safety net). The test for R5 therefore asserts the **observable difference**: that the mocked `getUpstreamBlocks` / `getBlockDependencies` is **not called** on a non-Python kernel, and that the notice is emitted — not the already-true "no error surfaced" assertion.

**KD-6 — failure categories ride a new `RunResult.failureCategory` string field, populated at two surfacing sites; exit codes are NOT subdivided.** PRD-002 commits the _falsifiable_ contract to "a stable machine-readable field in `--output json`" and leaves exit-code subdivision to ADR discretion. We keep the three existing `ExitCode` values and add a typed `failureCategory` to `RunResult`:

```ts
export type KernelFailureCategory =
  | "missing-kernel" // KernelNotRegisteredError (pre-flight)         → ExitCode.InvalidUsage (2)
  | "kernel-launch" // KernelLaunchError (POST/startNew failed)       → ExitCode.Error (1)
  | "kernel-died" // KernelDiedError (launch-time OR mid-run)       → ExitCode.Error (1)
  | "in-block"; // user code raised inside an executed block      → ExitCode.Error (1)
```

The critical correction over the prior draft: `buildMachineRunResult` (`run.ts:1212`) is **only reached on the success return path** (`run.ts:902-910`), after `engine.runProject` returns a summary. Three of the four classes are thrown _before_ that — `missing-kernel` (pre-flight inside `connect`), `kernel-launch` (`startNew` rejection), and launch-time `kernel-died` all propagate out of `engine.start()` and are caught by the **outer catch at `run.ts:522`**, which currently emits `outputJson({ success:false, error })` with no category. The field is therefore set at **two** sites:

- **Site (a) — outer catch (`run.ts:522-543`):** map the thrown error's `category` discriminant (each kernel error carries a literal `category` — KD-8) onto `failureCategory` and emit `outputJson({ success:false, error, failureCategory })` (and the `toon` equivalent). This is where `missing-kernel`, `kernel-launch`, and launch-time `kernel-died` are surfaced. The existing exit-code mapping (`run.ts:526-531`) is preserved: `missing-kernel` (a `KernelNotRegisteredError`) maps to `InvalidUsage` by adding it to that branch's union; the rest map to `Error`.
- **Site (b) — `buildMachineRunResult` (`run.ts:1212`):** for failures that _do_ flow through the per-block loop. Distinguishing `kernel-died`-mid-run from `in-block` requires a discriminant — **but the typed error class is destroyed before it reaches this function.** The engine's per-block catch stores the thrown `Error` _instance_ on `BlockExecutionResult.error` (`execution-engine.ts:410`), yet the CLI's `onBlockDone` callback (`run.ts:1048-1059`) immediately flattens it to a string when it builds the CLI-side `BlockResult`: `error: result.error?.message` (`run.ts:1058`). `BlockResult.error` is typed `string | undefined` (`run.ts:105`) and `RunExecutionState.blockResults` is `BlockResult[]` (`run.ts:193-194`); `buildMachineRunResult` receives that `BlockResult[]`. So an `instanceof KernelDiedError` against `blockResults[].error` can **never** be true — it would always be a string, and `kernel-died`-mid-run would collapse into `in-block`, violating R6. **The discriminant must be captured at the only point where the typed instance is still in scope — the `onBlockDone` serialization boundary (`run.ts:1058`):**
  1. Add `failureCategory?: KernelFailureCategory` to the CLI's `BlockResult` interface (`run.ts:97-106`). `RunExecutionState.blockResults` (`run.ts:193-194`) already holds `BlockResult[]`, so it carries the field automatically.
  2. In `onBlockDone` (`run.ts:1048-1059`), when `!result.success`, compute the discriminant from the still-typed `result.error` _before_ it is stringified — `result.error instanceof KernelDiedError ? 'kernel-died' : 'in-block'` — and push it as `failureCategory` alongside the existing `error: result.error?.message`. (Successful blocks leave it `undefined`.)
  3. Add `failureCategory?: KernelFailureCategory` to `RunResult` (`run.ts:156-164`). In `buildMachineRunResult`, when `summary.failedBlocks > 0`, read the discriminant **off the failing `BlockResult`** — `blockResults.find(b => !b.success)?.failureCategory ?? 'in-block'` — and set `result.failureCategory`. No `instanceof` is performed in `buildMachineRunResult`; it consumes the already-computed field.

  This requires `KernelClient.execute`/IOPub handling to throw a `KernelDiedError` (KD-8) on mid-run death rather than a bare `Error`, so the `instanceof` at step 2 (`run.ts:1058`, while the instance is live) sees the class. The `import { KernelDiedError }` lands in `run.ts`; `KernelFailureCategory` is imported alongside it from runtime-core (KD-8).

Each of the four classes maps to a distinct `failureCategory` value (no collapse → R6). The ADR-noted latent `InvalidUsage`-vs-`Error` overlap is resolved _on the new field_, not by inventing exit codes.

**KD-7 — idle/startup timeout becomes a configurable value.** ADR-002 flags the fixed `waitForKernelIdle(timeoutMs = 30000)` (`kernel-client.ts:97`) as a design-doc parameter: heavy kernels (IJulia precompile, JVM warmup) exceed 30s. ADR-002 asks for "configurable OR kernel-aware" — a design-doc parameter, satisfied by a `RuntimeConfig` field. We add `RuntimeConfig.kernelStartupTimeoutMs?: number` (default 30000, preserving today's behavior) threaded through `connect` into `waitForKernelIdle`. **The user-facing `--kernel-timeout` CLI flag is deferred** out of Phase 1: ADR-002 does not mandate it, the only Phase-1 integration kernel (`bash`) reaches idle sub-second and cannot exercise it, and shipping an untested user-facing knob adds surface and a doc/test obligation with no Phase-1 coverage. The `RuntimeConfig` field alone satisfies ADR-002 (byte-stable default, configurable for programmatic/MCP callers and a future heavy-kernel target). If a later phase adds a heavy-kernel integration target, the CLI flag lands then with a unit test asserting it threads through to `waitForKernelIdle`. A Phase-1 unit test still asserts that a non-default `kernelStartupTimeoutMs` on `RuntimeConfig` is forwarded into `waitForKernelIdle`, so the config field is not untested surface.

**KD-8 — kernel-failure errors are a typed family in runtime-core, each with a `category` discriminant.** `KernelNotRegisteredError` carries `{ requested: string, available: KernelspecSummary[] }` and a literal `category = 'missing-kernel'`; it renders "Install it … Installed kernels: …" (Scenario 2). `KernelLaunchError` (`category = 'kernel-launch'`) wraps the launch diagnostic (Scenario 6). `KernelDiedError` (`category = 'kernel-died'`) replaces the bare `'Kernel is dead'` _and_ is thrown on mid-run death so the class survives the per-block catch (KD-6 site b). The `category` discriminant is the single mechanism both surfacing sites read to set `failureCategory` (KD-6).

**KD-9 — the default vitest config gains an `exclude` glob; integration tests run in a separate project/script.** The root `vitest.config.ts` uses `include: ['**/*.test.ts']` with `bail: 1` and no `exclude`. A file named `*.integration.test.ts` matches that glob and **would** be collected by the default `pnpm test` (`vitest run`). Env-gating alone (a runtime `it.skipIf(!RUN_INTEGRATION_TESTS)`) still _collects_ the file and, with `bail: 1` and no local Python, risks interfering. Therefore the default config **does change** (correcting the prior "unchanged" claim): add `exclude: ['**/*.integration.test.ts']` to the default `vitest.config.ts`. Add a dedicated `vitest.integration.config.ts` that includes _only_ `**/*.integration.test.ts` and a `test:integration` script (`vitest run -c vitest.integration.config.ts`) invoked **solely** by the new `integration-kernels` CI job. `RUN_INTEGRATION_TESTS=true` remains a defense-in-depth runtime gate (so an accidental direct invocation without a provisioned venv self-skips), not the collection-exclusion mechanism. The always-on mocked `test` job is unaffected by the integration files because the `exclude` glob keeps them out of collection.

async connect(serverUrl: string, kernelName: string = 'python3'): Promise<string | undefined> {
  // ... existing ServerConnection / manager setup ...
  let language: string | undefined
  if (kernelName !== 'python3') {
    language = await this.preflightKernelspec(serverUrl, kernelName) // R2, KD-2/KD-3
  }
  try {
    this.session = await this.sessionManager.startNew({
      name: 'deepnote-cli', path: 'deepnote-cli', type: 'notebook',

  return language // engine stores it for the forward-looking predicate (KD-3)

/** GET /api/kernelspecs; throws KernelNotRegisteredError if absent. Returns the kernelspec language. */
private async preflightKernelspec(serverUrl: string, kernelName: string): Promise<string | undefined> {
  let specs: KernelspecsResponse
  try {
    specs = await fetchKernelspecs(serverUrl)           // GET {serverUrl}/api/kernelspecs
  } catch {
    return undefined                                    // R2: fall back to /api readiness, not a missing-kernel error
  }
  const spec = specs.kernelspecs[kernelName]
  if (!spec) {
    throw new KernelNotRegisteredError(kernelName, summarize(specs)) // R2: before any POST
  }
  return spec.spec.language
}

**Definition of done:** all degradation unit tests green in existing CI; the no-`_dntk`-dispatched assertion passes; both analyzer call sites are demonstrably skipped on non-Python; Python suites unchanged; `failureCategory` present and distinct for all four classes at both surfacing sites — specifically, the mid-run `KernelDiedError` test proves `'kernel-died'` does not collapse to `'in-block'` despite the `run.ts:1058` string-flattening (discriminant captured on `BlockResult.failureCategory`, read by `buildMachineRunResult`).

## Migration & Rollback

`kernelName` defaults to `'python3'` and `kernelStartupTimeoutMs` defaults to `30000`, so this is a **pure addition** with no migration step. Every existing call — `connect(url)` (one-arg, still valid via the default parameter; callers that ignore the new return value are unaffected), every `RuntimeConfig` without `kernelName`, the MCP `handleRun` path — resolves to exactly today's behavior: same kernelspec, same JSON-WebSocket negotiation, same idle-wait, same outputs, **and the pre-flight GET is skipped entirely** (KD-2). The Python execution path is therefore byte-stable; existing mocked suites pass unchanged with no edits beyond the additive new tests. The one default-config change is the vitest `exclude` glob (KD-9), which only removes not-yet-existing integration files from collection and does not alter any existing test's behavior. **Rollback** is reverting the additive commits; because no on-disk format, no schema, and no exit-code contract changed, rollback leaves no residue (no `language` field is written in Phase 1 — that is Phase 2). The only new always-present runtime behavior for non-Python runs is the one pre-flight GET against an already-running server, which is negligible against kernel startup (ADR-002 Consequences).

## Roadmap Connection

This design implements **PRD-002 Phase 1** under roadmap story **`m2/s5/alternative-kernels`**. It realizes the three Phase-1 prerequisite ADRs across their feature nodes:

- **`non-python-launch-model`** (ADR-002) → R1, R2, R7 (thread `kernelName`, pre-flight validation, real-kernel e2e).
- **`kernel-name-selection`** (ADR-003) → R3 (the `selectKernelName` resolver + `--kernel` flag; the notebook-declared tier is forward-declared for Phase 2 but not wired).
- **`degradation-behavior`** (ADR-004) → R4, R5 (value-add hard-fail, reactivity bypass).

Phase 1 deliberately stops short of: the notebook `language` schema field + import/export language preservation (PRD-002 **Phase 2**, which slots the declared tier under `--kernel` with no contract change), the `--list-kernels` discovery surface (PRD-002 **Phase 3**, which reuses this design's `/api/kernelspecs` pre-flight as its enumeration source _and_ the captured kernelspec `language` as a forward-looking signal), and the user-facing `--kernel-timeout` flag (deferred per KD-7 until a heavy-kernel integration target exists). The `DEEPNOTE_KERNEL` env tier remains a documented-but-unbuilt extension point (ADR-003).

## Open Questions

- **Per-language idle-timeout defaults.** KD-7 ships the configurable _shape_ (`RuntimeConfig.kernelStartupTimeoutMs`) with a flat 30s default. Should a kernel-aware default table (e.g. JVM/IJulia → 120s) and the user-facing `--kernel-timeout` flag land together once a real heavy-kernel integration target exists? (Non-blocking: `bash` never exercises the timeout.)
- **Toolkit version pin policy for the integration job.** Pin to an exact `deepnote-toolkit[server]` version vs. a floor (`>=`) — pin chosen for CI determinism; who owns the bump cadence?
- **Should `image/png` originate from a committed fixture or be generated by the bash fixture at run time?** Generating it exercises a fuller path (kernel → display_data → IOPub → decode); a committed expected-bundle makes the assertion stricter. Leaning generate-and-assert-MIME-type to keep R7 honest about the live transport.
- **`AstAnalyzerInternalError` base class.** Research notes it extends plain `Error`, not `DeepnoteError`. Phase 1 bypasses the analyzer entirely on non-Python (R5), and both analyzer call sites already swallow it today — so this is not load-bearing here. Should a follow-up reconcile it into the `DeepnoteError` family for consistent categorization? (Out of scope for Phase 1.)
